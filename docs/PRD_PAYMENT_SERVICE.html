01. 개요 및 배경

문제 정의

SolSQLD는 SQLD 자격증 시험 대비 및 SQL 실습 플랫폼으로, AI 해설·SQL 리뷰·학습 플랜 등의 AI 기능을 제공하고 있다. 현재 모든 사용자에게 일 3회의 무료 AI 사용 쿼타를 부여하고 있으나, 학습 몰입도가 높은 사용자에게는 이 제한이 학습 흐름을 끊는 주요 이탈 요인이 되고 있다.

제안

부분유료제(Freemium) 모델을 도입하여, 기본 기능(시험 응시, SQL 실습, 무한 연습)은 무료로 유지하되, AI 기능을 제한 없이 사용하고자 하는 사용자에게 월 2,900원의 프리미엄 구독을 제공한다.

목표 지표

지표목표측정 방법

해지율(Churn)월 10% 이하해지 수 / 기간 내 구독자 수

02. 현황 분석

이미 구축된 플랜 인프라

결제 연동만 빠져 있을 뿐, 플랜 관리 인프라는 이미 완성 상태이다. 추가 개발은 결제 플로우와 엔타이틀먼트 자동화에 집중할 수 있다.

테이블역할현재 상태
ai.plans 플랜 정의 (free / premium) free, premium 두 플랜 정의 완료. is_paid=true로 유료 구분
ai.user_entitlements 사용자별 플랜 부여 source_type에 'payment' 값 지원, 실제 결제 연동 없음

06. 결제 플로우

6-1. 최초 구독 (빌링키 발급 + 첫 결제)

1. 구독 버튼 클릭 → 2. 토스 결제창 → 3. 카드 인증 → 4. 빌링키 발급 → 5. 첫 결제 실행 → 6. 엔타이틀먼트 부여 
// 프론트엔드: 토스페이먼츠 SDK
const tossPayments = await loadTossPayments(clientKey);
await tossPayments.requestBillingAuth('카드', {
  customerKey: user.id,       // SolSQLD 유저 ID
  successUrl: `${BASE_URL}/api/billing/success`,
  failUrl: `${BASE_URL}/api/billing/fail`,
});

// 백엔드: /api/billing/success 콜백
// 1. authKey로 빌링키 발급 → POST /v1/billing/authorizations/issue
// 2. 빌링키로 첫 결제 → POST /v1/billing/{billingKey}
// 3. 결제 성공 시 → ai.user_entitlements INSERT (source_type='payment')
// 4. ai.model_routes에서 premium 라우팅 즉시 적용

6-2. 자동 갱신 (매월 반복)

1. 갱신일 도래 → 2. 스케줄러 트리거 → 3. 빌링키 결제 → 4. 웹훅 수신 → 5. 엔타이틀먼트 연장 
// 스케줄러 (cron 또는 APScheduler)
# 매일 09:00 KST — 만료 예정 구독 갱신 실행
async def renew_subscriptions():
    due = await get_subscriptions_due_today()
    for sub in due:
        result = await toss_billing_pay(sub.billing_key, amount=2900)
        if result.success:
            await extend_entitlement(sub.user_id, days=30)
            await record_payment(sub.id, result)
        else:
            await handle_payment_failure(sub, result)  # 재시도 로직

6-3. 해지

1. 해지 요청 → 2. 다음 갱신 취소 → 3. 잔여 기간 유지 → 4. 만료 시 Free 전환

해지 즉시 서비스가 중단되지 않으며, 이미 결제한 기간이 끝날 때까지 프리미엄 기능을 유지한다. 만료 후 자동으로 Free 플랜으로 전환되고, 일일 사용 쿼타가 3회로 복귀한다.

6-4. 결제 실패 재시도 정책

재시도 횟수간격액션
1차즉시첫 결제 실패 시 바로 1회 재시도
2차+24시간이메일 알림 발송 ("결제 수단을 확인해 주세요")

07. 백엔드 설계

7-1. 신규 DB 테이블

-- billing 스키마 생성
CREATE SCHEMA IF NOT EXISTS billing;

-- 구독 정보
CREATE TABLE billing.subscriptions (
    subscription_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES auth.users(user_id) ON DELETE CASCADE,
    plan_code       TEXT NOT NULL REFERENCES ai.plans(plan_code),
    billing_key     TEXT NOT NULL,           -- 토스 빌링키 (암호화 저장)
    status          TEXT NOT NULL CHECK (
        status IN ('active', 'cancelled', 'past_due', 'expired')
    ),
    current_period_start TIMESTAMPTZ NOT NULL,
    current_period_end   TIMESTAMPTZ NOT NULL,
    cancelled_at    TIMESTAMPTZ NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (user_id, plan_code)
);

-- 결제 이력
CREATE TABLE billing.payments (
    payment_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    subscription_id UUID NOT NULL REFERENCES billing.subscriptions(subscription_id),
    user_id       UUID NOT NULL REFERENCES auth.users(user_id) ON DELETE CASCADE,
    toss_payment_key TEXT NOT NULL UNIQUE,   -- 토스 paymentKey
    amount        INTEGER NOT NULL,          -- 원 단위 (2900)
    status        TEXT NOT NULL CHECK (
        status IN ('paid', 'failed', 'refunded', 'partial_refund')
    ),
    paid_at       TIMESTAMPTZ NULL,
    failed_reason TEXT NULL,
    receipt_url   TEXT NULL,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 결제 실패 재시도 추적
CREATE TABLE billing.retry_queue (
    retry_id       BIGSERIAL PRIMARY KEY,
    subscription_id UUID NOT NULL REFERENCES billing.subscriptions(subscription_id),
    attempt_number INTEGER NOT NULL DEFAULT 1,
    scheduled_at   TIMESTAMPTZ NOT NULL,
    attempted_at   TIMESTAMPTZ NULL,
    result         TEXT NULL CHECK (result IN ('success', 'failed')),
    error_detail   TEXT NULL,
    created_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

7-4. 엔타이틀먼트 연동

결제 성공 시 기존 ai.user_entitlements 테이블에 레코드를 생성하며, AI 서비스는 기존 로직을 그대로 사용한다. 추가 코드 변경 최소화가 핵심이다. 

async def activate_premium(user_id: str, subscription_id: str):
    """결제 성공 후 프리미엄 활성화"""
    now = datetime.now(timezone.utc)
    await db.execute("""
        INSERT INTO ai.user_entitlements
            (user_id, plan_code, source_type, source_reference, starts_at, ends_at)
        VALUES ($1, 'premium', 'payment', $2, $3, $4)
        ON CONFLICT (user_id, plan_code)
        DO UPDATE SET
            ends_at = $4,
            source_reference = $2,
            updated_at = now()
    """, user_id, subscription_id, now, now + timedelta(days=30))

    # premium 라우팅의 daily_limit을 무제한으로 변경
    # → model_routes에 premium 전용 행이 이미 존재하므로
    #   daily_limit=9999로 UPDATE (또는 unlimited 플래그 사용)

7-5. Premium daily_limit 변경

-- premium 플랜: daily_limit 30 → 9999 (사실상 무제한)
UPDATE ai.model_routes
SET daily_limit = 9999, updated_at = now()
WHERE plan_code = 'premium';
설계 원칙: 최소 변경 AI 서비스 레이어(service.py)는 이미 ai.model_routes에서 daily_limit를 읽어 쿼타를 관리하므로, 결제 연동 시 AI 서비스 코드 수정이 불필요하다. 결제 → 엔타이틀먼트 → 라우팅까지 데이터 레이어에서 완결된다.

09. 보안 및 법적 규정

9-1. 결제 보안

항목대응
카드 정보 보관 서버에 카드 번호 저장 안 함. 토스페이먼츠 빌링키만 암호화 저장 (AES-256)
웹훅 검증 토스 Webhook Secret으로 HMAC-SHA256 서명 검증 후 처리
API 인증 billing API는 기존 JWT 인증 미들웨어 적용. 토스 API 호출 시 Basic Auth (시크릿키)
CSRF 기존 CSRF 미들웨어 적용. 웹훅 엔드포인트는 HMAC 검증으로 대체하여 CSRF 면제
멱등성 토스 paymentKey를 billing.payments.toss_payment_key에 UNIQUE로 관리

9-2. 법적 고지 (전자상거래법)

필수 고지 사항 (전자상거래법 제13조)
  • 상품명: SolSQLD AI 프리미엄 (월간 구독)
  • 가격: 월 2,900원 (VAT 포함)
  • 결제 방식: 매월 자동 결제 (카드)
  • 구독 해지: 마이페이지 > 구독 관리에서 즉시 해지 가능
  • 환불 규정: 구독 개시일로부터 7일 이내 전액 환불, 이후 잔여일 비례 환불
  • 사업자 정보: 상호, 대표자, 사업자등록번호, 통신판매업 신고번호

11. 위험 요소 및 대응

위험수준영향대응
사업자 등록 지연 HIGH PG 가입 자체 불가 Week 1 시작 전 사업자 등록 완료. 간이과세자로 빠르게 진행
AI API 비용 초과 HIGH 프리미엄 사용자 급증 시 Claude API 비용이 구독 수익 초과 일 상한 9,999회이나 실 사용은 30~50회 예상. 모니터링으로 이상치 감지. 비용 임계치 초과 시 알림 + 일시 throttle 가능하도록 daily_limit 동적 조정
결제 실패율 높음 MID 해지율 증가 3회 재시도 정책 + 이메일/인앱 알림으로 결제 수단 업데이트 유도
통신판매업 미신고 MID 과태료 부과 가능 사업자 등록과 함께 통신판매업 신고 병행 (관할 구청, 온라인 신청)
환불 분쟁 LOW CS 부담 구독 페이지에 환불 정책 명시. 7일 이내 전액 환불 보장으로 분쟁 최소화
무료 → 유료 전환율 저조 MID 수익 미달 첫 7일 무료 체험(trial) 도입 검토. A/B 테스트로 전환 유도 위치/문구 최적화. 프로모션 코드(source_type='promotion') 활용
