> 작성일: 2026-04-04

## 1. 개요

### 요청 배경
1. 사용자가 자신의 정보를 확인/변경할 수 있는 마이페이지 필요
2. 아이디(이메일)/비밀번호 찾기 기능 필요
3. 개인정보 수집 동의를 받고 있으므로 회원 탈퇴 수단 필수

### 현재 상태
- 헤더 우측: `[닉네임 님 환영합니다!] [로그아웃]`
- 비밀번호/닉네임 변경 기능 없음
- 계정 찾기/복구 수단 없음
- 회원 탈퇴 기능 없음
- SMTP(이메일 발송) 인프라 없음

## 2. 마이페이지 (`/mypage`)

### 2-1. 진입점

헤더 우측 배치 변경:

```
[닉네임 님 환영합니다!]  [마이페이지 아이콘]  [로그아웃]
```

- 마이페이지 아이콘: lucide `Settings` 또는 `UserCog` 아이콘 (텍스트 없이 아이콘만, 호버 시 "마이페이지" 툴팁)
- 클릭 시 `/mypage`로 이동

### 2-2. 페이지 레이아웃

```
┌─────────────────────────────────────────────┐
│  마이페이지                                    │
│  내 계정 정보를 확인하고 관리할 수 있습니다.          │
├─────────────────────────────────────────────┤
│                                             │
│  ┌── 내 정보 ──────────────────────────┐     │
│  │ 이메일      qpo134@naver.com        │     │
│  │ 닉네임      alledeli    [변경]       │     │
│  │ 가입일      2026-03-28              │     │
│  │ 보유 포인트  120pt                   │     │
│  └────────────────────────────────────┘     │
│                                             │
│  ┌── 비밀번호 변경 ────────────────────┐     │
│  │ 현재 비밀번호  [••••••••]            │     │
│  │ 새 비밀번호    [••••••••]            │     │
│  │ 새 비밀번호 확인 [••••••••]          │     │
│  │              [비밀번호 변경]          │     │
│  └────────────────────────────────────┘     │
│                                             │
│  ┌── 약관 및 동의 ────────────────────┐     │
│  │ 서비스 이용약관       [보기]         │     │
│  │ 개인정보 수집 및 이용  [보기]         │     │
│  │ 동의일: 2026-03-28                 │     │
│  └────────────────────────────────────┘     │
│                                             │
│  ┌── 계정 삭제 ───────────────────────┐     │
│  │ 탈퇴 시 모든 학습 기록이 삭제됩니다.   │     │
│  │                   [회원 탈퇴]        │     │
│  └────────────────────────────────────┘     │
│                                             │
└─────────────────────────────────────────────┘
```

### 2-3. 기능 상세

#### A. 내 정보 섹션

| 항목 | 표시 | 수정 가능 |
|------|------|----------|
| 이메일 | `user.email` | 불가 (이메일 = 로그인 ID) |
| 닉네임 | `user.nickname` | 가능 — 인라인 편집 또는 모달 |
| 가입일 | `user.createdAt` | 불가 |
| 보유 포인트 | `user.points` | 불가 |

**닉네임 변경 흐름:**
1. [변경] 버튼 클릭 → 인라인 입력필드로 전환
2. 새 닉네임 입력 → [저장] / [취소]
3. 빈 값 불가, 최대 20자 제한
4. 저장 성공 시 헤더의 닉네임도 즉시 반영

#### B. 비밀번호 변경 섹션

| 필드 | 검증 규칙 |
|------|----------|
| 현재 비밀번호 | 필수, 백엔드에서 현재 비밀번호 일치 확인 |
| 새 비밀번호 | 필수, 최소 8자 |
| 새 비밀번호 확인 | 필수, 새 비밀번호와 일치 |

**흐름:**
1. 3개 필드 입력 → [비밀번호 변경] 클릭
2. 프론트 검증: 새 비밀번호 일치 확인, 최소 길이
3. 백엔드 검증: 현재 비밀번호 일치 확인
4. 성공 시: "비밀번호가 변경되었습니다" 알림, 필드 초기화
5. 실패 시: "현재 비밀번호가 일치하지 않습니다" 에러 표시

#### C. 약관 및 동의 섹션

- 서비스 이용약관 [보기]: 클릭 시 약관 전문 모달 (회원가입과 동일한 TERMS_TEXT)
- 개인정보 수집 및 이용 [보기]: 클릭 시 전문 모달 (PRIVACY_TEXT)
- 동의일 표시

#### D. 회원 탈퇴 섹션

**흐름:**
1. [회원 탈퇴] 클릭 → 확인 모달 표시
2. 모달 내용:
   ```
   정말 탈퇴하시겠습니까?

   탈퇴 시 아래 데이터가 모두 삭제되며 복구할 수 없습니다:
   · 학습 기록 (모의고사, SQL 실습)
   · 보유 포인트
   · 계정 정보

   확인을 위해 비밀번호를 입력해주세요.
   [비밀번호 입력]

   [취소]  [탈퇴하기]
   ```
3. 비밀번호 입력 + [탈퇴하기] 클릭 → 백엔드 처리
4. 성공 시: 로그아웃 + 메인 페이지로 이동 + "탈퇴가 완료되었습니다" 알림

## 3. 아이디/비밀번호 찾기

### 3-1. 현재 제약: SMTP 없음

이메일 발송이 불가하므로 일반적인 "이메일로 재설정 링크 발송" 방식은 사용 불가.

### 3-2. SMTP 없이 가능한 방식

| 방식 | 설명 | 장단점 |
|------|------|--------|
| **A. 본인 확인 후 즉시 재설정 (추천)** | 이메일 + 닉네임 + 가입일 조합으로 본인 확인 → 새 비밀번호 직접 설정 | 구현 간단, SMTP 불필요. 보안은 약하지만 학습 플랫폼 특성상 허용 가능 |
| B. 관리자 수동 처리 | "비밀번호를 잊으셨나요? solsqld@solsqld.com으로 문의하세요" | 자동화 없음, 관리 부담 |
| C. SMTP 먼저 구축 | 이메일 발송 인프라 셋업 후 정석 방식 | 정석이지만 추가 인프라 필요, 시간 소요 |

### 3-3. 추천: 방식 A — 본인 확인 후 즉시 재설정

**진입점:** 로그인 모달 하단에 "비밀번호를 잊으셨나요?" 링크 추가

**아이디(이메일) 찾기 흐름:**
```
1. "아이디를 잊으셨나요?" 클릭
2. 닉네임 입력
3. 백엔드에서 닉네임으로 검색 → 이메일 일부 마스킹 표시
   예: qpo***@naver.com
4. 사용자가 본인 이메일 확인
```

**비밀번호 재설정 흐름:**
```
1. "비밀번호를 잊으셨나요?" 클릭
2. 이메일 입력
3. 본인 확인 질문 표시:
   - 가입 시 등록한 닉네임을 입력해주세요: [입력]
4. 닉네임 일치 시 → 새 비밀번호 설정 화면
   - 새 비밀번호:     [••••••••]
   - 새 비밀번호 확인: [••••••••]
5. 변경 완료 → 로그인 모달로 이동
```

**보안 고려사항:**
- 닉네임이 공개 정보이므로 보안 수준은 낮음
- 단, SolSQLD는 학습 플랫폼이고 결제/민감 정보가 없으므로 허용 가능한 수준
- 향후 SMTP 구축 시 이메일 인증 방식으로 업그레이드 권장
- 비밀번호 재설정 시도 횟수 제한 필요 (5회/시간) — 무차별 대입 방지

### 3-4. UI 위치

로그인 모달(`AuthModal.tsx`)에 링크 추가:

```
┌──────────────────────────┐
│  로그인                    │
│  [이메일 입력]             │
│  [비밀번호 입력]           │
│                           │
│  [로그인 버튼]             │
│                           │
│  아이디를 잊으셨나요?       │  ← 새로 추가
│  비밀번호를 잊으셨나요?     │  ← 새로 추가
│                           │
│  계정이 없으신가요? 회원가입  │
└──────────────────────────┘
```

클릭 시 모달 내에서 화면 전환 (별도 페이지 아님, 모달 내 step 전환).

## 4. 필요한 백엔드 API

### 신규 엔드포인트

| # | Method | Path | 설명 |
|---|--------|------|------|
| 1 | `GET` | `/api/auth/profile` | 마이페이지 내 정보 조회 (이메일, 닉네임, 가입일, 포인트, 동의일) |
| 2 | `PUT` | `/api/auth/nickname` | 닉네임 변경 |
| 3 | `PUT` | `/api/auth/password` | 비밀번호 변경 (현재 비밀번호 검증 필수) |
| 4 | `DELETE` | `/api/auth/account` | 회원 탈퇴 (비밀번호 검증 필수, 관련 데이터 전체 삭제) |
| 5 | `POST` | `/api/auth/find-email` | 아이디 찾기 (닉네임 → 마스킹된 이메일 반환) |
| 6 | `POST` | `/api/auth/reset-password` | 비밀번호 재설정 (이메일 + 닉네임 검증 → 새 비밀번호 설정) |

### API 상세

#### 1. GET /api/auth/profile
```
Authorization: Bearer {token}

Response 200:
{
  "email": "qpo134@naver.com",
  "nickname": "alledeli",
  "createdAt": "2026-03-28T...",
  "points": 120,
  "termsAgreedAt": "2026-03-28T...",
  "privacyAgreedAt": "2026-03-28T..."
}
```

#### 2. PUT /api/auth/nickname
```
Authorization: Bearer {token}
Body: { "nickname": "새닉네임" }

Response 200: { "message": "닉네임이 변경되었습니다.", "nickname": "새닉네임" }
Response 400: { "detail": "닉네임은 1~20자로 입력해주세요." }
```

#### 3. PUT /api/auth/password
```
Authorization: Bearer {token}
Body: {
  "current_password": "...",
  "new_password": "..."
}

Response 200: { "message": "비밀번호가 변경되었습니다." }
Response 400: { "detail": "현재 비밀번호가 일치하지 않습니다." }
Response 400: { "detail": "새 비밀번호는 8자 이상이어야 합니다." }
```

#### 4. DELETE /api/auth/account
```
Authorization: Bearer {token}
Body: { "password": "..." }

Response 200: { "message": "회원 탈퇴가 완료되었습니다." }
Response 400: { "detail": "비밀번호가 일치하지 않습니다." }
```

삭제 대상 (CASCADE 순서):
1. `amplitude.raw_events` WHERE user_id = ?
2. `logs.*` WHERE user_id = ?
3. `memo.exam_attempt_memos` (attempt_id 기준)
4. `exam.exam_attempt_*` WHERE user_id = ?
5. `practice.sql_practice_attempts` WHERE user_id = ?
6. `dashboard.user_*` WHERE user_id = ?
7. `auth.user_consents` WHERE user_id = ?
8. `auth.users` WHERE user_id = ?

#### 5. POST /api/auth/find-email
```
Body: { "nickname": "alledeli" }

Response 200: { "maskedEmail": "qpo***@naver.com" }
Response 404: { "detail": "일치하는 계정을 찾을 수 없습니다." }
```

#### 6. POST /api/auth/reset-password
```
Body: {
  "email": "qpo134@naver.com",
  "nickname": "alledeli",
  "new_password": "..."
}

Response 200: { "message": "비밀번호가 재설정되었습니다." }
Response 400: { "detail": "계정 정보가 일치하지 않습니다." }
```
Rate limit: 5회/시간 (IP 기준)

## 5. 필요한 프론트엔드 작업

| 순서 | 작업 | 상세 |
|------|------|------|
| 1 | 헤더 마이페이지 아이콘 추가 | `[환영합니다!]` 과 `[로그아웃]` 사이에 아이콘 버튼, `/mypage`로 이동 |
| 2 | `MyPage.tsx` 페이지 생성 | `/mypage` 라우트 추가, 4개 섹션 구현 |
| 3 | 닉네임 변경 UI | 인라인 편집 + API 연동, AuthContext의 user 상태 즉시 업데이트 |
| 4 | 비밀번호 변경 폼 | 3개 입력필드 + 프론트 검증 + API 연동 |
| 5 | 회원 탈퇴 모달 | 확인 모달 + 비밀번호 입력 + API 연동 → 로그아웃 처리 |
| 6 | AuthModal에 찾기 링크 추가 | 로그인 모달 하단에 "아이디/비밀번호를 잊으셨나요?" |
| 7 | 아이디 찾기 화면 | 모달 내 step 전환, 닉네임 입력 → 마스킹 이메일 표시 |
| 8 | 비밀번호 재설정 화면 | 모달 내 step 전환, 이메일+닉네임 → 새 비밀번호 설정 |

## 6. 이벤트 로그 (EVENT_TAXONOMY 추가)

| Event Name | Trigger | Properties |
|------------|---------|------------|
| `mypage_viewed` | 마이페이지 진입 | `user_id` |
| `mypage_nickname_changed` | 닉네임 변경 성공 | `user_id`, `old_nickname`, `new_nickname` |
| `mypage_password_changed` | 비밀번호 변경 성공 | `user_id` |
| `mypage_account_deleted` | 회원 탈퇴 완료 | `user_id` |
| `auth_find_email_submitted` | 아이디 찾기 시도 | `nickname`, `found: boolean` |
| `auth_reset_password_submitted` | 비밀번호 재설정 시도 | `email_domain`, `success: boolean` |

## 7. 작업 우선순위 및 의존성

```
Phase 1: 마이페이지 (먼저)
  [back_dev] API 4개: profile, nickname, password, account
      ↓
  [front_dev] 헤더 아이콘 + MyPage.tsx + 라우트 추가
      ↓
  [qa_dev] 닉네임 변경/비밀번호 변경/탈퇴 검증

Phase 2: 계정 찾기 (이후)
  [back_dev] API 2개: find-email, reset-password + rate limit
      ↓
  [front_dev] AuthModal 내 아이디/비밀번호 찾기 step 추가
      ↓
  [qa_dev] 검증: 정상 흐름 + rate limit + 잘못된 정보 입력
```

## 8. 제외 항목 (다음 단계)

| 항목 | 제외 이유 |
|------|-----------|
| 이메일 변경 | 이메일 = 로그인 ID, 변경 시 본인 인증 필요 → SMTP 구축 후 |
| 프로필 이미지 | 현재 UI에 프로필 이미지 사용처 없음 |
| SMTP 기반 비밀번호 재설정 | 인프라 미구축. 향후 구축 시 현재 닉네임 검증 방식을 이메일 링크 방식으로 교체 |
| 로그인 이력 조회 | logs.auth_events에 데이터 있지만 사용자 노출 우선순위 낮음 |
